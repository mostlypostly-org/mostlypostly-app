Two targeted edits:

**Edit A — calendar.js, SortableJS animation (diagnosed ~line 285):**
Find the Sortable.create() call inside the DOMContentLoaded block (or wherever SortableJS initializes per `.calendar-day-cell`). Change:
  animation: 150
to:
  animation: 0

This eliminates the SortableJS init animation that causes newly-approved pills to visually shift on page reload. The drag ghost already provides drag feedback — the animation parameter is for item enter/leave transitions during initialization, not during the drag itself.

**Edit B — calendar.js, Remove href for manager_approved posts (diagnosed ~line 392):**
Find the Remove link rendered for manager_approved posts in the day panel action HTML. It currently reads:
  href="/manager/cancel-post?post=<postId>" (or equivalent template literal)
Append &return=calendar so it becomes:
  href="/manager/cancel-post?post=${post.id}&return=calendar"

**Edit C — manager.js, GET /cancel-post handler (diagnosed ~line 745-753):**
Find the GET /cancel-post route handler. After the post is cancelled (or near the redirect at the bottom), replace the unconditional redirect:
  return res.redirect(`/manager?salon=${salonId}`);
with a return-aware redirect:
  const returnTo = req.query.return === 'calendar' ? '/manager/calendar' : `/manager?salon=${salonId}`;
  return res.redirect(returnTo);

Do not change any other part of the cancel-post handler. The pattern matches /manager/approve (line 699), /manager/post-now, /manager/deny (line 847), and /manager/retry-post — all of which already handle return=calendar.
